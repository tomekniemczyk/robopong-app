🎯 Kalibracja — 3-etapowy proces

Rotation → Height → Oscillation + Speed. Pełny flow oryginału z komendami R/U/O/Q/W.

⚠ Kluczowe odkrycie: kalibracja jest 3-etapowa (3 osobne ekrany w UI), nie pojedyncza. Każdy etap reguluje inną oś robota.
⚠ Firmware NIE persystuje kalibracji! Wartości zapisane są tylko lokalnie (SharedPreferences / NSUserDefaults / Configuration.xml). Po każdym nowym połączeniu MUSI być re-wysłana.

Memory: feedback_calibration_required.md

1. Trzy etapy kalibracji

1

Rotation

Regulacja kąta rotacji głowicy.

Ball: zerowe silniki
H=50%, Osc=50%, Rot=0°
→ R{raw:000}\r

RotCAL = raw − 150

2

Height

Regulacja wysokości głowicy.

Ball: zerowe silniki
H=50%, Osc=50%, Rot=0°
→ U{raw:000}\r

HeightCAL = raw − 150

3

Oscillation + Speed

Oscylacja + prędkość silnika (z testowym rzutem).

Gen2: Top=73%, H=80%, Osc=50%
Gen1: Top=77%, H=80%, Osc=50%
→ V\r × 2, Q, B, O × 2, H, W

OscCAL = raw − 150, SpeedCAL = raw_top − baseline

6. SpeedCAL — algorytm z baseline

Źródło: ANDROID_APP_RE.md:715-736

def set_speed_cal(ball):
    robot_ball = advanced_ball_converter.get_robot_ball(ball, left_handed)
    # robot_ball.TopMotorSpeed to raw (28..210)
    # np. dla 73% → 28 + 182×0.73 = 161

    if robot_version == SecondRun:         # Gen2
        offset = robot_ball.TopMotorSpeed - 161  # baseline 161
    else:                                   # Original / Gen1
        offset = robot_ball.TopMotorSpeed - 170  # baseline 170
        # UWAGA: 170 a nie 168 (prawdopodobnie hardkodowany bug,
        # zaokrąglenie historyczne)

    config["SpeedCAL"] = offset  # zapis lokalny (może być 0/ujemny/dodatni)

    if offset > 0:
        robot.send(f"Q{offset:03d}\r")  # timeout 500ms
    # jeśli offset <= 0: TYLKO zapis lokalny, BEZ wysłania do robota
🐛 BUG w oryginale: SpeedCAL jest martwym kodem! Wartość odczytana z config w setBall(), ale natychmiast nadpisywana na 0 przed obliczeniem PWM. Jest wysyłany tylko raz przez komendę Q, nigdy nie wpływa na kolejne komendy B/A.

Źródło: BUSINESS_LOGIC_COMPLETE.md:503-510

7. Pseudokod — complete() etap 3

def complete_calibration(ball):
    # a) przywróć left-handed (w etapie 3 wymuszamy IsLeftHanded=false)
    is_left_handed = _saved_is_left_handed

    # b) Obniż height o 30 punktów (baseline offset compensation)
    ball.head_position.height -= 30

    # c) SetSpeedCAL
    set_speed_cal(ball)               # wysyła Q{offset} jeśli offset > 0

    # d) Zeruj silniki i wyślij piłkę
    ball.top_motor_speed = 0
    ball.bottom_motor_speed = 0
    set_ball(ball, force=True, is_cal=True)
    wait(500)

    # e) Adjust Oscillation × 2 (z 500ms przerwą)
    adjust_oscillation(ball)          # O{raw}, OscCAL = raw - 150
    wait(500)
    adjust_oscillation(ball)          # powtórka!

    # f) FinishCalibration
    config["IsCalibrated"] = True
    clear_ball()                      # H\r
    set_adjustment()                  # W000\r

11. Jak działa AcePad obecnie

  • Tylko 1 ekran kalibracji (nie 3) — uproszczony UI.
  • Persystencja w SQLite, per-device (addr PK).
  • Apply calibration po każdym connect (OK, zgodne z FW).
  • Sekwencja rzutu: set_ball → 300ms → T → 1500ms → H — silniki STOP po rzucie.
  • Brak auto-rozgrzewki przy wejściu w kalibrację.

Odstępstwo od oryginału: brak 3-etapowego flow. Pytanie do uzgodnienia — czy wprowadzić, czy zostawić uproszczony model?

12. Obszary do weryfikacji

  • Delay 500ms po V #1 — RE Windows wymienia, iOS/Android RE nie. Do sprawdzenia na żywo.
  • Height − 30 w complete() — co to dokładnie robi fizycznie? Baseline offset? Dlaczego akurat 30?
  • Dlaczego O #2? — komenda adjust_oscillation wywołana 2× z 500ms przerwą. Potrzeba potwierdzenia.
  • Baseline 170 dla Gen1 — matematycznie powinno być 168 (dla 77%). Hardkodowany bug czy celowe zaokrąglenie?
  • Spacja przed LEDs w komendzie B — przykłady pokazują B0663029700150142150 6\r ze spacją, ale format string C# nie ma spacji. Weryfikacja.
