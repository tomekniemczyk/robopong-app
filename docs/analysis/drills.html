🎾 Drille — SYNC vs ASYNC

Dwa tryby wysyłania drilla, wybierane automatycznie na podstawie firmware i transportu.

🔀 Auto-detect trybu: oryginał wybiera tryb automatycznie. Użytkownik nie ma przełącznika — decyzja należy do kodu biznesowego.

1. Dwa tryby — przegląd

SYNC

B + T per piłka

Warunek: FW < 701 || USB_Mode

  • Aplikacja steruje każdym rzutem
  • Każda piłka: osobne B\r + T\r
  • Aplikacja zarządza timingiem (WaitTime)
  • Kompensacja opóźnień (throw > 625ms)
ASYNC

wTA + A × N + END

Warunek: FW ≥ 701 && BLE

  • App ładuje wszystkie piłki do robota
  • Robot sam zarządza rzutami i timingiem
  • App tylko czeka na K per rzut, N na koniec
  • Brak kompensacji — robot optymalizuje

2. Decyzja trybu — drzewo wyboru

run_drill(drill):
  ├─ transport == USB?
  │    ├─ YES → SYNC (USB_Mode=true wymusza stary protokół)
  │    └─ NO ↓
  ├─ firmware < 701?
  │    ├─ YES → SYNC
  │    └─ NO ↓
  └─ ASYNC (BLE + FW ≥ 701)

  Konsekwencja: Windows = zawsze SYNC (FTDI USB, brak BLE)
               Android USB = zawsze SYNC
               Android BLE + FW ≥ 701 = ASYNC
               iOS BLE + FW ≥ 701 = ASYNC

3. Format komendy B / A (setBall)

B (stary) i A (nowy) mają identyczny layout — różnica tylko w prefiksie i momencie wykonania rzutu:

B{dirTop:1}{|topPWM|:3}{dirBot:1}{|botPWM|:3}{osc:3}{height:3}{rot:3} {leds:1}\r A{dirTop:1}{|topPWM|:3}{dirBot:1}{|botPWM|:3}{osc:3}{height:3}{rot:3} {leds:1}\r Przykład: B0663029700150142150 6\r = Top forward PWM=663, Bot forward PWM=297, Osc=150, H=142, Rot=150, LEDs=6
PoleDługośćOpis
dirTop / dirBot1 cyfra0 = forward, 1 = reverse (bit kierunku)
|topPWM| / |botPWM|3 cyfrywartość absolutna PWM po przeliczeniu z raw
osc, height, rot3 cyfry każdezero-padded RAW values
leds1 cyfra0–8 (spin indicator)

PWM formula

# Standard (FW != 206, dzisiejsze roboty)
pwm = raw_motor × 4.016

# Legacy (FW == 206)
pwm = raw_motor + 5

# Przykład: TopMotor = 73%
raw = 28 + 182 × 0.73 = 161
pwm = 161 × 4.016 = 646.576 → 646 (int konwersja)
Uwaga: przykłady w RE pokazują spację przed LEDs (... 150 6\r), ale format string C# nie ma spacji. Do weryfikacji na żywym robocie.

10. Pełny pseudokod — run_drill

def run_drill(drill):
    # --- Setup ---
    clear_ball()                      # H\r
    wait(200)
    clear_ball()                      # H\r
    wait(1000)

    if fw < 701 or usb_mode:          # SYNC
        set_ball(first_ball, force=True)
        wait(300)
        set_ball(first_ball, force=True)
        wait(1500)

    # --- Run ---
    if fw < 701 or usb_mode:
        # TRYB SYNC
        count = 0
        while drill_running and count < drill.Count:
            for ball in drill.Balls:
                set_ball(ball, force=True)       # B{params}
                wait(ball.WaitTime)
                throw_start = now()
                throw_ball()                     # T → K (max 7s)
                count += 1
                if now() - throw_start > 0.625:
                    # kompensacja
                    next_wait_factor = 0.000125 × ball.WaitTime + 0.5625
            if drill.Delay > 0 and count < drill.Count:
                set_ball(first_ball)
                wait(drill.Delay × 1000)
        clear_ball()
    else:
        # TRYB ASYNC
        for ball in drill.Balls:
            send(f"wTA{ball.WaitTime // 10:03d}\r")
            wait_response(max=5000)
            wait(500)
            send(format_A(ball))
            wait_response(max=5000)
            wait(250)

        seed = random.randint(1, 255)
        send(f"END{drill.Count:03d}{seed:03d}{drill.Delay * 10:03d}\r")

def set_ball(ball, force, is_cal=False):
    # Skip jeśli !force && cached == ball
    if not force and cached_ball.ballNumber == ball.ballNumber:
        return

    dir_top = 1 if ball.top_motor < 0 else 0
    dir_bot = 1 if ball.bottom_motor < 0 else 0

    # BUG: SpeedCAL zawsze 0 (martwy kod!)
    num3 = config["SpeedCAL"] if fw < 221 else 0
    num3 = 0  # ← nadpisanie!

    if fw == 206:
        pwm_top = abs(ball.top_motor) + 5
        pwm_bot = abs(ball.bottom_motor) + 5
    else:
        pwm_top = abs(ball.top_motor) × 4.016 + num3
        pwm_bot = abs(ball.bottom_motor) × 4.016 + num3

    leds = get_leds(ball.top_motor, ball.bottom_motor)

    # Decyzja: SYNC vs ASYNC (poziom setBall!)
    if fw < 701 or is_cal or usb_mode:
        send(f"B{dir_top}{int(pwm_top):03d}{dir_bot}{int(pwm_bot):03d}"
             f"{ball.osc:03d}{ball.height:03d}{ball.rot:03d} {leds}\r")
        wait_response(max=500)
    else:
        send(f"wTA{ball.wait_time // 10:03d}\r")
        wait_response(max=5000)
        wait(500)
        send(f"A{dir_top}{int(pwm_top):03d}{dir_bot}{int(pwm_bot):03d}"
             f"{ball.osc:03d}{ball.height:03d}{ball.rot:03d} {leds}\r")
        wait_response(max=5000)

    cached_ball.ballNumber = ball.ballNumber

11. LEDs formula (spin indicator)

Źródło: BUSINESS_LOGIC_COMPLETE.md:270-280, ANDROID_APP_RE.md

def get_leds(top_raw, bottom_raw):
    diff = top_raw - bottom_raw
    ratio = abs(diff) / 360.0

    if ratio == 0:       return 0  # None
    if ratio <= 0.10:    return 5 if diff > 0 else 1  # OneTop / OneBottom
    if ratio <= 0.50:    return 6 if diff > 0 else 2  # TwoTop / TwoBottom
    if ratio <= 0.75:    return 7 if diff > 0 else 3  # ThreeTop / ThreeBottom
    return 8 if diff > 0 else 4                       # FourTop / FourBottom
Sprzeczność: protocol-flow.html:697-708 ma inne progi (0.08/0.17/.../0.7) — prawdopodobnie przestarzałe. Autorytatywna jest wersja 10/50/75/75+ z BUSINESS_LOGIC.

14. Random Mode (Y + P) — tylko ASYNC BLE FW≥701

Źródło: ANDROID_APP_RE.md:614-642, BUSINESS_LOGIC_COMPLETE.md:512-524

Komenda Y (część 1)

Y{numBallsLow:2}{numBallsHigh:2}{numBallsType:1} {topMotorLow:2}{topMotorHigh:2}{topMotorType:1} {bottomMotorLow:2}{bottomMotorHigh:2}{bottomMotorType:1} {heightLow:3}{heightHigh:3}\r

Komenda P (część 2)

P{heightType:1}{oscLow:2}{oscHigh:2}{oscType:1} {rotLow:2}{rotHigh:2}{rotType:1} {waitHigh/10:3}{waitLow/10:3}{waitType:1} {dirTopHigh:1}{dirTopLow:1}{dirBotHigh:1}{dirBotLow:1}{dirRotHigh:1}{dirRotLow:1}\r

RandomType: 1 = Range (losuj z zakresu), 2 = EitherOr (50/50), 3 = None (stały). Wartości 100 clampowane do 99.

15. Znane bugi w oryginale

🐛 BUG #1 — SpeedCAL martwy kod
Offset SpeedCAL odczytywany z config w setBall(), ale natychmiast nadpisywany na 0. Wpływa tylko na komendę Q, nigdy na B/A.

BUSINESS_LOGIC.md:503-510

🐛 BUG #2 — Random ball clamp
Warunek BottomMotorHigh == 100 sprawdzany 2× zamiast BottomMotorLow == 100 w drugim branchu.

BUSINESS_LOGIC.md:512-518

🐛 BUG #3 — Random ball P-command direction
Kierunek dla BottomMotorLow w komendzie P używa TopMotorLow (skopiowany błędny field).

BUSINESS_LOGIC.md:519-524

🐛 BUG #4 — RobotBall.Equals ignoruje WaitTime
Dwie piłki z różnym WaitTime uznawane za identyczne (cache skip na _currentBall.ballNumber == ball.ballNumber), więc zmiana samego WaitTime nie triggeruje nowego setBall.

16. Jak działa AcePad obecnie

  • Przełącznik drill_mode w UI (sync/async) — w oryginale nie ma!
  • Auto-detect firmware zaimplementowany (patrz memory project_drill_modes.md)
  • Własny warmup 3s (patrz warmup.html)

Odstępstwo od oryginału: ręczny przełącznik trybu. Do uzgodnienia — czy zostawić, czy usunąć na rzecz pełnego auto-detect?

17. Obszary do weryfikacji

  • Spacja przed LEDs w B/A — RE przykłady ze spacją, format string bez. Testy na żywym robocie.
  • LEDs progi — rozjazd między HTML (0.08/0.17/...) a BUSINESS_LOGIC (10/50/75/75+).
  • Kompensacja throw > 625ms — dokładny moment zastosowania współczynnika.
  • USB\r signal — wysyłana przez Android przy init USB. Czy Windows też?
  • Minimalny delay między chunkami BLE — RE podaje 200ms. Potwierdzenie.
