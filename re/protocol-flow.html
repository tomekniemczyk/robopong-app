Przegląd przepływu

Główna sekwencja od włączenia do treningu:

1. Połączenie
BLE: skan urządzeń "NEWGY"/"NWGY" → GATT → MLDP service
USB: FTDI 9600 baud → 0x5A × 2 → 115200 baud
↓
2. Inicjalizacja
Z→ Z→ Z→ H→ K
Czekaj max 20s na odpowiedź K
↓
3. Detekcja firmware
F→ 701 (3-cyfrowy numer ASCII)
FW ≥ 701 → nowy protokół (A/wTA/END), asynchroniczne rzuty
↓
4. Kalibracja
Ustawienie TopMotor, Height, Oscillation → komendy V, B, T, U, O, R
Wynik: SpeedCAL, OscCAL, RotCAL, HeightCAL
↓
5. Drill / Trening
FW < 701: B → delay → T (synchronicznie, pilot steruje)
FW ≥ 701 BLE: wTA + A × N → END (robot steruje)

Kluczowe wartości (nie zmieniaj!)

Centrum głowicy = 150 (oscylacja, rotacja, height bazowy) Oscylacja: zakres 127–173, centrum 150 Height: zakres 75–210 Rotacja: zakres 90–210, centrum 150 Prędkość: raw × 4.016 = wartość PWM w komendzie B/A BLE terminator: \r (0x0D) USB terminator: \r\n (0x0D 0x0A) — Android USB: \r only!

Kalibracja

3-etapowa kalibracja z oryginalnych aplikacji. AcePad upraszcza to do jednego ekranu.

Etap 1: Rotacja

Begin Calibration
V\r
× 2 (500ms między) — wchodzi w tryb kalibracji
↓
Ustaw piłkę kalibracyjną
Height=50%, Osc=50%, Rot=0, TopMotor=0%, BottomMotor=0%
→ B000000000015014215008\r
↓
User adjusts rotation
Strzałki: ±1 (ciągły kierunek), ±2 (zmiana kierunku)
Każda zmiana → B...{rot}...\r
↓
Zapisz
RotCAL = raw - 150
R{raw:000}\r np. R153\r

Etap 2: Wysokość

User adjusts height
Strzałki ±1 → B...{height}...\r
↓
Zapisz
HeightCAL = raw - 150
U{raw:000}\r np. U142\r

Etap 3: Prędkość + Oscylacja

Piłka kalibracyjna
Gen2: TopMotor=73%, Bot=0%, Height=80%, Osc=50%, Rot=0 → raw top=161
Gen1: TopMotor=77%, Bot=0%, Height=80%, Osc=50%, Rot=0 → raw top=168
↓
Wyślij piłkę — silniki kręcą
B0646000000150183150 6\r
top=161×4.016=646, bot=0, osc=150, h=183, rot=150
↓
User adjusts
↑↓ TopMotorSpeed ±1%
←→ Oscillation ±2 (Win) / ±4 (Android/iOS)
Każda zmiana → B...{updated}...\r
↓
Test throw
T\r → K (max 7s timeout)
↓
Done — zapisz kalibrację
a) Height -= 30 (baseline offset!)
b) SpeedCAL = raw - baseline (161 Gen2 / 170 Gen1)
   Jeśli offset > 0: Q{offset:000}\r
c) Zeruj silniki: B0000000000150142150 0\r
d) OscCAL = osc_raw - 150: O{raw:000}\r × 2 (500ms)
e) Finish: H\r + W000\r
AcePad defaults (Gen2): top=160, bot=0, osc=150, h=183, rot=150, wait=1000ms.
Kalibracja per-device (keyed by BLE address w tabeli SQLite calibration).

Drill — wykonanie

Setup (wspólny)

Clear
H\r → 200ms → H\r → 1000ms
↓
FW < 701: Pre-load first ball
B{first_ball}\r → 300ms → B{first_ball}\r → 1500ms

FW < 701 lub USB — synchroniczny

Pilot steruje każdym rzutem.

Loop: dla każdej piłki
B{ball_params}\r → K
wait(WaitTime ms)
T\r → K (max 7s)
drillBallCount++
↓ powtórz × count
Delay między cyklami
Jeśli delay > 0: B{first_ball}\r + wait(delay × 1000ms)
↓
Cleanup
H\r — ClearBall
Kompensacja wolnych rzutów: jeśli throw > 625ms, następny wait *= (0.000125 × waitTime + 0.5625)

FW ≥ 701 + BLE — asynchroniczny

Robot sam zarządza rzutami.

Załaduj wszystkie piłki
Dla każdej piłki:
wTA{wait/10:000}\r → K
A{ball_params}\r → K
wait 250ms → następna
↓
Execute
END{count:3}{seed:3}{delay×10:3}\r
np. END050127080 = 50 piłek, seed=127, delay=8s
↓
Robot rzuca autonomicznie
K po każdym rzucie (count++)
N = drill zakończony
↓
Cleanup
H\r

Przykład: 3 piłki, FW ≥ 701

Advanced / Manual Mode

Pojedyncza piłka z pełną kontrolą parametrów.

Ustawienie piłki

Parametry wejściowe
TopMotorSpeed: -100% do +100% (ujemne = reverse)
BottomMotorSpeed: -100% do +100%
Height: 0–100%
Oscillation: 0–100% (0=prawo, 50=środek, 100=lewo)
Rotation: -90° do +90°
WaitTime: 500–2000ms
↓ konwersja
Obliczenie wartości raw
Motor raw = 28 + 182 × (percent / 100)
B-value = raw × 4.016
Height raw = 75 + 135 × (percent / 100)
Osc raw = 173 - 46 × (percent / 100)
Rot raw = 150 + 60 × (degrees / 90)
↓
Wyślij

Formuły konwersji

Motor Speed (% → raw → B-command value)

raw = 28 + 182 × (percent / 100) // percent > 0 raw = -(210 × |percent| / 100) // percent < 0 (reverse) raw = 0 // percent = 0 B-value = |raw| × 4.016 // firmware != 206 B-value = |raw| + 5 // firmware == 206 (legacy) -- Tabela konwersji -- 0% → raw= 0 → B= 0 50% → raw=119 → B= 478 10% → raw= 46 → B= 185 73% → raw=161 → B= 646 (Gen2 baseline) 25% → raw= 73 → B= 293 77% → raw=168 → B= 675 (Gen1 baseline) 100% → raw=210 → B= 843

Height (% → raw)

raw = 75 + round(135 × percent / 100) 0% → 75 (najniżej) 50% → 142 80% → 183 (domyślna kalibracja) 100% → 210 (najwyżej)

Oscillation (% → raw)

raw = 173 - round(46 × percent / 100) // praworęczny raw = 173 - round(46 × (100-percent) / 100) // leworęczny (mirror) 0% → 173 (skrajnie prawo) 50% → 150 (centrum) 100% → 127 (skrajnie lewo)

Rotation (degrees → raw)

raw = 150 + round(60 × degrees / 90) -90° → 90 (pełno lewo) -45° → 120 0° → 150 (prosto) 45° → 180 90° → 210 (pełno prawo)

LEDs (spin indicator)

ratio = |topMotor_raw - botMotor_raw| / 360 ratio = 0 → 0 (NoSpin) ratio < 0.08 → 1 (OneTop) ratio < 0.17 → 2 (OneBottom) ratio < 0.225 → 3 (TwoBottom) ratio < 0.28 → 4 (ThreeBottom) ratio < 0.4 → 5 (FourBottom) ratio < 0.55 → 6 (TwoTop) ratio < 0.7 → 7 (ThreeTop) ratio ≥ 0.7 → 8 (FourTop)

Format komendy B/A

B{dirTop:1}{|topSpeed|:3}{dirBot:1}{|botSpeed|:3}{osc:3}{height:3}{rot:3} {leds:1}\r dirTop/dirBot: 0 = forward, 1 = reverse topSpeed/botSpeed: 3 cyfry zero-padded (B-value po ×4.016) osc/height/rot: 3 cyfry zero-padded (raw values) leds: 1 cyfra 0-8 (spacja przed!) Przykład: B0663029700150142150 6\r B|0|663|0|297|150|142|150| 6| d top d bot osc h rot led

Bugi w oryginalnym kodzie Newgy

1. SpeedCAL zawsze zerowany
BUSINESS_LOGIC_COMPLETE.md:503-510
Wartość SpeedCAL jest odczytywana z config, a potem natychmiast ustawiana na 0. Nigdy nie jest dodawana do obliczeń prędkości — jedynie wysyłana przez komendę Q.
2. Random ball — BottomMotorHigh sprawdzany 2×
BUSINESS_LOGIC_COMPLETE.md:512-518
W warunku clamping, BottomMotorHigh jest sprawdzany zamiast BottomMotorLow w drugiej gałęzi.
3. Random ball P-command — zły kierunek
BUSINESS_LOGIC_COMPLETE.md:519-524
Kierunek (direction) dla BottomMotorLow w komendzie P bierze wartość z TopMotorLow zamiast BottomMotorLow.
4. RobotBall.Equals ignoruje WaitTime
Dwie piłki z różnym WaitTime są traktowane jako identyczne — porównanie nie uwzględnia tego pola.
